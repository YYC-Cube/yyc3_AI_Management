## Overview

YYC³ API Management 平台使用 GitHub Actions 进行持续集成和持续部署 (CI/CD)。工作流旨在确保代码质量、运行测试并自动化部署。

## Workflows

### 1. CI/CD Pipeline (`ci-cd.yml`)

这是主工作流，在推送到 `main` 和 `develop` 分支以及针对这些分支的 Pull Request 时运行。

**Jobs:**

- **test**: 使用 PostgreSQL 和 Redis 服务运行所有后端测试
  - Lint 代码 (非阻塞)
  - 执行类型检查 (非阻塞)
  - 运行数据库迁移
  - 执行单元测试并生成覆盖率报告
  - 运行缓存、AI 和 WebSocket 测试 (非阻塞)
  - 执行安全扫描
  - 运行集成测试 (非阻塞)
  - 上传覆盖率报告到 Codecov
  
- **build**: 构建前端和后端
  - 构建 Next.js 前端
  - 构建后端 TypeScript 代码
  - 上传构建产物

- **docker**: 构建并推送 Docker 镜像 (仅在 `main` 分支)
  - 使用 Docker Buildx 进行多平台构建
  - 推送到 Docker Hub 并打上正确标签

- **deploy**: 部署到生产环境 (仅在 `main` 分支)
  - 执行零停机滚动更新
  - 运行健康检查
  - 执行冒烟测试
  - 发送 Slack 通知

- **preview**: 为 PR 创建 Vercel 预览部署
  - 部署到 Vercel 预览环境
  - 创建包含预览 URL 的 check run

**Key Features:**

- 对非关键测试使用 `continue-on-error: true`，允许流水线即使某些测试失败也能完成
- 提供全面的测试覆盖，涵盖不同的测试套件
- 实现带健康检查的零停机部署
- 与外部服务集成 (Codecov, Docker Hub, Vercel, Slack)

### 2. Test Gate (`test-gate.yml`)

在多个 Node.js 版本上运行的全面测试工作流。

**Jobs:**

- **test**: 在 Node.js 18.x 和 20.x 上运行
  - 类型检查代码
  - Lint 代码
  - 运行带覆盖率的单元测试
  - 检查覆盖率阈值 (最低80%，非阻塞)
  - 运行集成测试 (非阻塞)
  - 使用 Playwright 运行 E2E 测试 (非阻塞)
  - 上传测试产物和报告

- **security-scan**: 执行安全审计
  - 运行 npm audit (非阻塞)
  - 执行质量门禁检查

- **performance-test**: 运行性能测试 (仅在 `main` 分支)
  - 使用 k6 进行负载测试

- **deployment-ready**: 标记构建为可部署状态
  - 仅在 `main` 分支运行
  - 需要其他所有 job 通过

**Key Features:**

- 跨多个 Node.js 版本的矩阵测试
- 带详细报告的全面测试覆盖
- 使用 npm audit 进行安全扫描
- 使用 k6 进行性能测试
- 上传测试产物用于调试

### 3. Chromatic (`chromatic.yml`)

UI 组件的视觉回归测试。

**Jobs:**

- **chromatic**: 发布到 Chromatic
  - 执行视觉回归测试
  - 在 `main` 分支自动接受更改
  - 即使有视觉更改也以零退出码退出

## Configuration Requirements

### GitHub Secrets

需要在 GitHub 仓库中配置以下 Secrets：

#### For CI/CD Pipeline:
- `OPENAI_API_KEY_TEST`: 测试用 OpenAI API 密钥
- `NEXT_PUBLIC_API_BASE_URL`: 前端 API 基础 URL
- `DOCKER_USERNAME`: Docker Hub 用户名
- `DOCKER_PASSWORD`: Docker Hub 密码
- `PRODUCTION_SSH_KEY`: 生产部署 SSH 密钥
- `PRODUCTION_USER`: 生产服务器用户名
- `PRODUCTION_HOST`: 生产服务器主机名
- `SLACK_WEBHOOK`: Slack Webhook URL 用于通知
- `SNYK_TOKEN`: Snyk 认证令牌 (可选)
- `VERCEL_TOKEN`: Vercel 部署令牌 (可选)
- `VERCEL_ORG_ID`: Vercel 组织 ID (可选)
- `VERCEL_PROJECT_ID`: Vercel 项目 ID (可选)
- `VERCEL_SCOPE`: Vercel 范围 (可选)

#### For Chromatic:
- `CHROMATIC_PROJECT_TOKEN`: Chromatic 项目令牌

### Environment Variables

工作流使用以下环境变量：

- `NODE_VERSION`: Node.js 版本 (默认: 18.x)
- `DOCKER_IMAGE`: Docker 镜像名称 (默认: yyc3-api-management)

## Workflow Triggers

- **Push**: 在 `main` 和 `develop` 分支上运行
- **Pull Request**: 在针对 `main` 和 `develop` 分支的 PR 上运行

## Best Practices

1. **Non-blocking Tests**: 可能因环境问题而失败的测试（缓存、AI、WebSocket）设置为 `continue-on-error: true`
2. **Coverage Thresholds**: 测试门禁检查80%代码覆盖率，但为非阻塞
3. **Zero-downtime Deployments**: 使用滚动更新和健康检查确保无服务中断
4. **Artifact Retention**: 测试结果和构建产物保留7-30天用于调试
5. **Security**: 定期执行安全审计（非阻塞）

## Troubleshooting

### Workflow Failures

如果工作流失败，请检查以下内容：

1. **Test Failures**: 查看 GitHub Actions UI 中的测试日志
2. **Build Failures**: 检查 TypeScript 错误或依赖问题
3. **Deployment Failures**: 验证服务器访问和健康检查端点
4. **Coverage Issues**: 查看覆盖率报告识别未测试代码

### Missing Secrets

如果看到关于缺少 Secrets 的错误：

1. 进入仓库 Settings > Secrets and variables > Actions
2. 按上述列表添加所需的 Secrets
3. 重新运行失败的工作流

## Local Testing

在运行工作流之前，可以使用健康检查脚本验证本地环境设置：

```bash
npm run workflow-check
```

这将检查：
- Node.js 和 npm 版本
- 项目结构和依赖项
- 配置文件 (TypeScript, ESLint 等)
- GitHub Actions 工作流文件
- 环境配置
- Docker 和 Docker Compose
- 必需服务 (PostgreSQL, Redis)

要在本地测试工作流，可以使用 [act](https://github.com/nektos/act)：

```bash
# 安装 act
brew install act  # macOS

# 运行特定工作流
act -j test

# 带 secrets 运行
act -j test --secret-file .secrets
```

## Updating Workflows

更新工作流时：

1. 在功能分支中测试更改
2. 使用 GitHub 的工作流编辑器审查工作流语法
3. 确保所有必需的 Secrets 都已记录
4. 更新此文档以反映任何更改
5. 通过创建 Pull Request 来测试工作流

## Maintenance

- **Weekly**: 审查安全审计结果
- **Monthly**: 更新 Node.js 和 action 版本
- **Quarterly**: 审查和优化工作流性能
- **As needed**: 更新 Secrets 和凭证
